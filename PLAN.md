# 实时交易反欺诈 + 客户画像 平台规划（PLAN）

> 技术栈：Java 生态（Spring Boot + Flink + Drools + Spark/MLlib + MLeap）
> 目标：贴近生产的完整工程（高可用、规则热更新、模型可迭代、可监控）
> 关系：**客户画像是特征供给方，实时反欺诈是特征消费方**，两者共享特征层。

---

## 0. 总览架构

```
                  ┌─────────────────────────────────────┐
                  │        共享特征 / 画像平台             │
                  │  在线特征(Redis/HBase) + 离线(Hive)    │
                  │       标签元数据(MySQL)               │
                  └───────────────┬─────────────────────┘
                                  │ 提供特征 (feature-client)
        交易请求 → fraud-gateway(同步, ≤50ms 返回风险等级)
                                  ↓
              拉特征 ─┬→ 随机森林评分(MLeap) → fraudScore
                      └→ Drools 规则集 → 命中信号 + 硬规则结论
                                  ↓
                       决策表统一裁决(放行/拦截/复核/加验证)
                                  ↓
                  决策落库 → 离线分析(Hive) → 反哺规则 & 重训模型
```

核心非功能要求：**低延迟（同步链路 P99 ≤ 50ms 向银行返回风险等级，硬 SLA）**、高可用、规则与模型均可热更新、可审计、可监控。详见 §3.6。

## 2. 客户画像项目

### 2.1 标签体系分层

| 类型 | 说明 | 例子 |
|------|------|------|
| 统计类标签 | 直接统计聚合 | 近30天交易笔数、月均余额、最常用渠道 |
| 规则类标签 | 阈值/规则判定 | 高净值客户、活跃用户、流失预警 |
| 算法/挖掘类标签 | 模型预测 | 客户分群(聚类)、信用评分、欺诈倾向 |

时效维度：
- **离线标签**（T+1，数仓批处理）：人口属性、长周期行为
- **实时标签**（流计算）：当前登录设备、当日交易累计、近 1 小时行为（反欺诈强依赖）

### 2.2 特征数据来源

1. 业务库：账户、交易流水、登录日志（CDC/Binlog 采集，Canal/Debezium）
2. 埋点行为数据：App/网银点击、停留、操作路径
3. 设备与环境：设备指纹、IP、地理位置
4. 三方数据：征信、黑名单、运营商（合规前提下）
5. 离线数仓加工结果：宽表派生指标

采集链路（**Kafka 为中央解耦枢纽**，一份数据多路并行消费）：
```
银行接入方(各 sourceId) ─推送交易报文→ ┐
业务库 ─Binlog(Canal/Debezium)→        ├→ Kafka (流量解耦 + 削峰)
                                        ┘
                                          │
        ┌─────────────────────────────────┼─────────────────────────────────┐
        ▼                                 ▼                                 ▼
  Flink(实时特征)                  Logstash(清洗)                     Sink → Hive/Iceberg
   → Redis/HBase(在线特征)           → ES(检索/案件调查/分析)            → Spark 批加工 → 离线标签 → Redis
```

链路说明：
- **Kafka**：流量解耦 + 削峰，银行各接入方(sourceId)推送的交易报文与业务库 Binlog 都汇入，下游多消费者并行
- **Flink → Redis**：实时特征，反欺诈低延迟链路用
- **Logstash → ES**：Logstash 从 Kafka 消费 → filter 清洗/解析/脱敏 → 写入 ES（存**已清洗**数据）。ES 定位为交易流水全文检索、案件调查/人工复核、近实时分析，**非实时拦截存储**
- **Hive/Iceberg**：离线数仓 / 模型训练
- 注：交易报文入口与 3.5 的「交易来源(Source)」呼应——每个接入方注册 sourceId 并推送至 Kafka

#### 2.2.1 Kafka 削峰与扩展（高并发）

**先厘清：同步决策请求不走 Kafka。** 走 Kafka 的只有「交易数据流」（异步，供特征/ES/离线）；银行要 50ms 拿风险等级的「同步决策请求」走 gateway → engine **直连 RPC**——扔进 Kafka 就无法同步返回，会破坏 50ms SLA。两条路物理分离。

| 路径 | 走 Kafka? | 说明 |
|------|-----------|------|
| 同步决策请求(50ms 返回风险等级) | 否 | gateway→engine 直连，不依赖 Kafka 排空进度 |
| 交易数据流(特征/ES/离线) | 是 | 异步、高吞吐、削峰 |

**Kafka 扛高并发的机制：**
- **分区(Partition)水平扩展**（根本）：topic 拆 N 分区分布到多 broker，并行度 = 分区数；加分区/加 broker 即扩容
- **顺序写磁盘 + PageCache**：append-only 顺序 IO，借 OS 页缓存
- **零拷贝(sendfile)**：PageCache 直发网卡，省 CPU/内存带宽
- **批量 + 压缩**：producer 攒批(`batch.size`+`linger.ms`) + lz4/zstd
- **消费者组并行消费**：分区分给组内多消费者，消费者数 ≤ 分区数线性扩容（Flink/Logstash 即此）

**削峰填谷（核心价值）**：Kafka 不追求"实时处理完"，而是先用磁盘 retention「接住」洪峰，下游(Flink/ES)按稳定速率匀速消费、不被打垮；瞬时 10x 洪峰被缓冲，数据不丢。**同步决策链路完全不依赖 Kafka 排空进度，故洪峰冲击数据流时银行侧 50ms 决策不受影响**——这正是数据流/决策流分离的深层原因。

**两个坑**：
- **数据倾斜**：分区键按账号/卡号 hash，既保同账号有序又均匀分散；避免会聚集的 key
- **背压与 lag**：盯 consumer lag，持续涨即消费能力不足→加分区+消费者；Flink 自带背压反压上游

**容量估算**：目标 TPS ÷ 单分区吞吐 → 分区数，预留 2~3x 余量应对洪峰。

### 2.3 存储分层

| 用途 | 存储 | 说明 |
|------|------|------|
| 在线实时读取(低延迟) | Redis / HBase | 按 userId 查, 毫秒级 |
| 离线宽表 / 标签明细 | Hive / Iceberg | 全量、可回溯、供分析与训练 |
| 标签元数据管理 | MySQL | 标签定义、口径、版本、生命周期 |
| 实时特征中转 / 解耦 | Kafka | 流计算输入、中央解耦枢纽 |
| 交易检索 / 案件调查 | Elasticsearch | Logstash 清洗后写入，全文检索与多维查询（非实时拦截存储） |
| (进阶) 统一特征平台 | Feast | 解决训练/服务特征一致性 |

原则：**在线 + 离线双存储**。离线算全量保准确，实时算增量保时效，在线层(Redis)统一对外提供 `getFeatures(userId, featureKeys)`。

### 2.4 画像 ↔ 反欺诈：双向数据流与冷启动

两个项目的真正耦合点在数据流向，需厘清三件事：

**(1) 原始交易流水是画像最大的单一数据源，但它是「原始数据」，非反欺诈产出。**
反欺诈与画像**共用同一份原始交易流水**（业务库 Binlog → Kafka），是平级消费方，不存在"画像吃反欺诈数据"。

**(2) 画像数据远不止交易**：还有行为埋点、账户属性、设备指纹、三方数据；交易流水量最大但只是其一。

**(3) 反欺诈产出的数据量小但价值极高，反向回流给画像**：

```
原始交易流水 ──→ 画像(算特征) ──→ 反欺诈(规则+模型评分)
                  ▲                      │
                  │                      ▼
                  └──── 反欺诈产出 ◄──── 决策结果/命中记录/案件标签
                        (回流为: 风险类标签 + 模型训练样本 label)
```

反欺诈产出的「近30天被拦截N次」「风险等级」「确认欺诈/正常的案件标签」回流进画像变成**风险类标签**，同时作为随机森林的**训练 label**。

**(4) 冷启动（鸡生蛋）问题 → 决定里程碑顺序**：
模型需要画像特征 + 案件标签才能训练，但系统初上线无标签。因此**先用规则(M1~M5)跑起来拦截、积累案件标签，再训模型(M6)**。这就是模型层排在规则之后的根本原因。

### 2.5 ES 索引设计（大数据量）

核心：**时间滚动 + 冷热分层 + mapping 精简**。

- **时间滚动索引 + ILM**：按天/月建索引（`txn-2026.05.29`），ILM 做 rollover（按 50GB / 时长 / 文档数触发）与生命周期 hot(SSD) → warm(HDD) → cold → delete；监管留存多年靠分层沉到廉价存储
- **分片策略**：单分片 30~50GB，避免过度分片；副本 1 保高可用
- **mapping 精简**（决定存储成本与查询速度）：
  - 精确匹配字段（卡号、账号、sourceId、状态码、交易码）用 `keyword`，不用 text
  - 金额用 `long` 存「分」，避免浮点精度问题
  - 不查询字段 `index: false`，不聚合字段关 `doc_values`
  - 时间字段统一 `date` + format
- **读写别名**：写别名指向当前索引，读别名跨多时间索引，应用层只认别名
- **老索引优化**：只读后 `force merge` 减 segment + `best_compression`
- **设计驱动**：按案件调查查询模式（账号/卡号/时间范围/金额区间/状态）优先保证可查可聚合

### 2.6 Logstash 清洗与脱敏

**清洗内容（filter 阶段）：**
- **解析结构化**：原始交易报文(JSON/分隔符/定长) → 结构化字段（`json`/`dissect`/`grok`/`kv`）
- **标准化**：时间时区与格式统一(`date`)、金额单位统一(分↔元)、编码映射(交易码/渠道码→枚举)
- **补维(enrich)**：关联维表补机构名/地区名/商户名（`translate`/`jdbc_streaming`/`elasticsearch` filter）
- **类型转换 + 去重过滤**：`mutate convert`；丢弃心跳/测试/无效记录
- **字段裁剪**：`remove_field` 减体积

**数据脱敏（金融重点，落 ES 前完成）：**

| 敏感字段 | 脱敏方式 | 说明 |
|----------|----------|------|
| 卡号、账号 | 掩码(masking) | 保留前6后4，中间打码（`gsub`/`ruby`） |
| 身份证、手机号、姓名、地址 | 掩码 | 部分遮蔽 |
| 需关联统计的字段 | **HMAC 哈希(带盐)** | 同卡号→同 hash，保留可聚合/可关联但不可逆（`fingerprint`） |

关键设计点：
- **HMAC 一致性哈希是精髓**：脱敏后同卡号仍哈希成同值，ES 里可按脱敏卡号聚合、查"该卡所有交易"，但拿不到明文——案件调查与画像统计都靠它
- **双链路区分**：ES(检索/调查) = 脱敏版；实时风控计算(Flink→Redis)若需完整字段走另一条受控链路（明文加密存储或同用 HMAC 关联）。**脱敏不能破坏风控计算所需字段**

### 2.7 用户画像生成算子目录

算子(Operator) = 输入字段 + 计算逻辑 + 输出特征名，可复用可配置。同一算子需**离线(Spark/HiveSQL)与实时(Flink)两套实现且口径一致**。

| 算子类别 | 算子 | 示例 |
|----------|------|------|
| 统计聚合 | count / sum / avg / max / min / **distinct count** | 近30天交易笔数、月均余额、去重对手方数 |
| 时间窗口 | 滑动/滚动窗口聚合、首次/最近一次、时间间隔 | 当日累计金额、距上次交易天数、近1h次数 |
| 比率/占比 | ratio、同比/环比、增长率 | 跨行转账占比、交易额环比增长 |
| 排序/Top | 众数(mode)、TopN、排名 | 最常用渠道、Top5 常用商户 |
| 趋势/序列 | 趋势(升/降)、波动性、序列模式 | 余额趋势、消费波动性 |
| RFM | Recency / Frequency / Monetary | 客户价值分层 |
| 关系/图 | 关联设备数、共用IP数、一度/二度关联账户 | 团伙欺诈识别(图计算) |
| 地理 | 常用登录地、异地判定、距离 | 异地交易标记、不可能行程 |
| 分箱/打标 | 阈值分段(binning)、if-else 标签映射 | 高净值客户、活跃度等级 |
| 缺失/异常处理 | 填充、归一化/Z-score、离群值处理 | 模型特征预处理 |
| 编码 | one-hot、label encoding、类别归并 | 模型类别特征 |
| 模型类 | 聚类分群(KMeans)、模型评分、异常检测分 | 客户分群、欺诈倾向分 |
| 衍生 | 表达式计算、多特征组合 | 自定义派生指标 |

依赖侧重：反欺诈强依赖去重计数、时间窗口、地理、关系/图算子；画像营销侧强依赖 RFM、排序/Top、趋势、分箱打标。

## 3. 实时反欺诈项目

### 3.1 场景维度模型

「渠道」与「业务类型」是两个正交维度，规则按维度组合命中，**不按场景硬编码**：

| 维度 | 取值 |
|------|------|
| 渠道(channel) | 手机银行 App、网银 PC、ATM、柜面、第三方支付 |
| 业务类型(bizType) | 转账、汇款、消费、缴费、贷款放款 |
| 对手方 | 行内/跨行、是否首次交易、是否黑名单 |

每条规则带 `scope`（适用渠道/业务类型），运行时先按场景过滤候选规则集再执行。新增场景 = 补规则，不改引擎。

### 3.2 规则分层

- **黑白名单层**（最快，命中直接出结果）：黑卡、黑设备、黑 IP
- **阈值/统计规则层**：单笔限额、日累计、N 分钟内交易次数/金额
- **关联/序列规则层**（Flink CEP）：短时多地登录、试探性小额后大额转账
- **模型评分层**：随机森林输出欺诈概率分

### 3.3 规则引擎（Drools）

- 规则用 **DRL + 决策表(Excel/CSV)** 双形式：复杂逻辑写 DRL，阈值类用决策表交业务方维护
- **热加载**：规则存 DB/Nexus，`fraud-rule-mgmt` 审核通过后推送，KieContainer/KieScanner 不重启生效
- **Fact 模型**：`TransactionEvent`(交易上下文) + `FeatureSnapshot`(画像实时特征) + `RuleResult`(输出)
- **分层与优先级**：用 `agenda-group` 按场景/层分组 + `salience` 控优先级；运行时只激活对应场景 agenda-group，减少匹配开销
- **性能**：进引擎前一次性从 Redis 批量取齐特征，规则里不做 IO

### 3.4 时序规则（Flink CEP）

`fraud-cep` 处理事件序列/时间窗口类规则（如「5 分钟内 3 次失败登录后成功再转账」），作为 Drools 的补充。

### 3.5 来源 ↔ 规则集 绑定模型（规则配置与生效解耦）

平台运营流程：**先配规则 → 规则归入规则集 → 交易来源绑定规则集**。规则"写一次、到处绑"，交易来源通过绑定表选择性命中规则集，而非把渠道条件硬编码进每条规则。

```
规则(Rule) ──多对多── 规则集(RuleSet) ──绑定── 交易来源(Source)
   单条逻辑            一组规则打包          接入方注册(sourceId)
```

- **交易来源(Source)**：每个上游接入系统注册 `sourceId/接入码`，如「手机银行-转账」「网银-汇款」「第三方支付-消费」
- **规则集(RuleSet)**：一组规则打包（如"高风险转账规则集""新设备规则集"）
- **绑定关系**：交易来源绑定一个或多个规则集；交易带 `sourceId` 进来，引擎路由到对应规则集执行

价值：规则可复用；新来源接入只做绑定不动规则；只跑相关规则集（性能）；每个来源跑哪些规则清晰可审计。

**落到 Drools**：一个规则集 ≈ 一个 `agenda-group`（或独立 KieBase）。运行时 gateway 拿 `sourceId` → 查绑定 → 只激活对应 agenda-group。

**数据模型（存 MySQL，归 fraud-rule-mgmt）**：

| 表 | 含义 |
|------|------|
| `t_source` | 交易来源(接入方) |
| `t_rule` | 规则定义(条件+动作+优先级) |
| `t_rule_set` | 规则集 |
| `t_rule_set_rule_rel` | 规则集-规则 多对多 |
| `t_source_ruleset_bind` | 来源-规则集 绑定（"绑定交易来源"那一步） |
| `t_decision_flow` | (可选)决策流/决策表，编排多规则集的裁决 |

### 3.6 性能与低延迟保障（P99 ≤ 50ms 硬 SLA）

50ms 是 P99/P999 目标（非平均值），银行侧超时即触发其自身兜底。核心原则：**同步链路只做内存计算，IO/重计算全部前置或异步化；宁可降级也不超时。**

**延迟预算分解：**

| 环节 | 预算 | 说明 |
|------|------|------|
| 网络往返(银行↔网关) | ~8ms | 同城/同机房部署是关键 |
| 反序列化 + 校验 | ~1ms | |
| 特征获取(Redis) | ~8ms | 最大瓶颈，须一次取齐 |
| Drools 规则执行 | ~3ms | 只跑绑定规则集 |
| MLeap 模型评分 | ~3ms | 进程内单条微秒级 |
| 决策聚合 + 序列化 | ~1ms | |
| 预留缓冲(GC/抖动) | ~10ms | 长尾余量 |

**优化手段（按层）：**
1. **特征获取**：全部预计算放 Redis，同步链路不现算；一笔交易所需特征用**一次 MGET/Pipeline/Lua** 取回（杜绝 N 次 round-trip）；多级缓存——黑白名单/规则元数据/模型放本地 Caffeine(L1)，用户特征走 Redis(L2)
2. **规则引擎**：预编译 KieBase + KieSession 池化（不重建）；用 agenda-group 只激活该 sourceId 绑定的规则集；规则内零 IO
3. **模型评分**：MLeap runtime 进程内嵌入 fraud-engine，免 RPC 跳转；模型常驻内存预热
4. **全链路异步化**：决策日志、ES 写入、案件回流、埋点 → 先返回银行再异步落库（Disruptor/异步 Kafka）
5. **JVM/GC**：用 ZGC/Shenandoah 把暂停压到亚毫秒（G1 几十 ms 暂停会吃穿预算）；大缓存堆外；启动预热 JIT

**兜底降级（保 SLA 的关键）**：每步设硬超时 + 熔断(Resilience4j)，任一依赖超时即走降级、立即返回：
- Redis 特征超时 → 部分特征/本地缓存兜底 → 偏保守风险等级
- 模型不可用 → 退化为纯规则决策
- 规则引擎异常 → 返回默认风险等级(中, 需复核)

→ 即使后端抖动，银行侧永远能在 50ms 内拿到风险等级。

**监控**：盯 P99/P999 而非平均值；分环节(feature/rule/model)埋点追踪长尾来源。

## 4. 模型层：决策树 + 随机森林

### 4.1 角色

| 模型 | 角色 |
|------|------|
| 随机森林 | 主力评分，输出欺诈概率分 + 特征重要性 |
| 单棵决策树 | 可解释兜底（路径即规则） |

### 4.2 训练 → 服务路线（Spark MLlib + MLeap）

```
Hive 特征宽表 + 案件标签
        ↓ Spark 读取
   VectorAssembler 特征拼装 + 样本均衡(weightCol/下采样)
        ↓
   DecisionTreeClassifier  (可解释兜底)
   RandomForestClassifier  (主力, CrossValidator 调 numTrees/maxDepth/maxBins)
        ↓ 离线评估(AUC, KS, 召回@固定误杀率)
   Spark ML Pipeline(特征处理 + 模型) 整体
        ↓ MLeap 序列化
   model.zip (入对象存储/MySQL 记版本)
        ↓ MLeap runtime(纯 JVM, 不依赖 SparkContext) 加载
   fraud-model-serving: 单条预测毫秒级, 支持版本热切换
```

要点：
- **整条 Pipeline（特征变换 + 模型）一起导出**，保证线上与训练特征处理一致，避免 training-serving skew
- 样本不平衡（欺诈 < 1%）用 `weightCol` 类别加权
- 训练宽表口径必须与线上 Redis 实时特征口径一致

### 4.3 双轨决策（模型分 + Drools）

进引擎前算好 `fraudScore` 塞进 Fact → 决策表裁决：

| 条件 | 决策 |
|------|------|
| 硬规则/黑名单命中 | 直接拦截（模型不参与） |
| fraudScore > 0.9 | 拦截 |
| 0.6 < score ≤ 0.9 | 加验证 / 人工复核 |
| score ≤ 0.6 且无规则命中 | 放行 |

模型推理在进引擎前完成，规则里只 `eval(fraudScore > 阈值)`，不做模型计算，保证延迟。

> 备注：「决策树 → Drools 规则反向导出」能力暂不做，先跑通随机森林评分主链路。

### 4.4 模型管理与监控

- 版本管理：model.zip 带版本号入库，灰度加载与热切换
- 离线评估：AUC / KS / 混淆矩阵 / 召回@固定误杀率
- 线上监控：特征分布漂移(PSI)、模型分分布、命中率
- 重训练：定期用新案件回流样本重训
