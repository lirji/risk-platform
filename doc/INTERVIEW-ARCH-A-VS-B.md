# 面试速记卡：架构 A（离线评级）vs 架构 B（实时拦截）

> 配套详解见 [ARCHITECTURE-COMPARISON.md](ARCHITECTURE-COMPARISON.md)（全文）。

## 🎯 主线

> **同一风控平台的两条互补主线：B 是实时拦截（逐笔同步、50ms 出决策、Drools+模型），A 是离线评级（任务驱动 Spark 批、配置即数据、给客户打标）；不是替代，是在线层 + 离线层，靠 ES/特征衔接。**

## 一表看清

| 维度 | 架构 A 离线评级 | 架构 B 实时拦截 |
|------|----------------|----------------|
| 范式 | 批处理，Spark on YARN | 实时同步，常驻服务 |
| 触发 | 平台建任务→MySQL→Spark 执行 | 银行逐笔调 REST |
| 时延 | 分钟~小时 | P99 ≤ 50ms |
| 引擎 | 评分规则在 Spark 里批量打标 | Drools 请求时逐笔裁决 |
| 模型 | MySQL 配置 + Spark 内打分 | PMML + jpmml 进程内推理 |
| 配置 | **MySQL 统一存模型/规则/任务**（改库不改代码） | 内存绑定 + DRL/PMML 文件 |
| 数据中心 | **ES**（标签源+风险库） | **Redis**（在线特征）+ ES（检索） |
| 产出 | 评分/评级（打标，多维风险） | 逐笔拦截决策（放行/复核/拒绝） |
| 对应项目 | 客户画像 + 评分评级 | 实时交易反欺诈 |

## 为什么大数据引擎一个用 Spark、一个用 Flink/Redis

- **A 用 Spark**：扫全量历史（90天聚合、RFM、聚类评分），跨 Hive+ES 关联，可重算回溯——批处理强项
- **B 用 Flink/Redis**：当日累计、5分钟频次必须流式毫秒级，Spark 批延迟扛不住 50ms
- 本质 = **标签分层**：离线重型标签 Spark，实时窗口标签 Flink，覆盖不同时效区间（详见 [ARCHITECTURE-COMPARISON.md §6](ARCHITECTURE-COMPARISON.md)）

## 两者怎么拼成完整平台（衔接点）

1. **离线 → 在线**：A 算的客户评级/标签（ES/Hive）下沉同步到 B 的 Redis，成为实时规则/模型输入特征
2. **在线 → 离线**：B 的决策结果/案件标签回流，成为 A 模型重训的标注样本（冷启动与回流）
3. **ES 复用**：A 的风险库 + B 的检索库可统一为同一 ES 集群的不同索引

## "配置即数据"（A 的灵魂，常考）

- 模型、评分规则、评级阈值、任务全存 MySQL（`t_rating_task`/`t_score_rule`/`t_rating_grade`）
- Spark 引擎运行时 JDBC 自主读取 → **改库即改评分逻辑，不改代码不发版**
- 规则随闭包广播到 executor，`parallelize.map` 分布式打分

## 易追问点

| 追问 | 答 |
|------|-----|
| 为啥不都做成实时？ | 历史全量聚合/模型训练实时算不动且没必要；评级是 T+1 决策，不需要毫秒 |
| 为啥不都做成离线？ | 交易要当场放不放行，离线分钟级延迟钱早走了，反欺诈失效 |
| 两套规则引擎重复？ | 形态不同：A 批量打标、B 逐笔裁决；可解释口径对齐，但执行机制本就该分 |
| 落地到哪了？ | B 已 7 模块跑通；A 核心引擎 + Hive 宽表(存算分离)已落地实跑，仅两个 Tomcat Web UI 未做 |

## 🏁 收尾

> A 和 B 是离线层与在线层，覆盖"批量评级"和"实时拦截"两类风控诉求；用 Spark/Flink 分别匹配各自时效，靠 ES 和特征/标签双向衔接，构成"离线沉淀 → 在线决策 → 结果回流"的闭环。

## 关键词
